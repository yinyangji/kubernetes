本模块围绕三个核心抽象组织：
- `ImageManager`（`image_manager.go` + `types.go`）：对外入口，实现 `EnsureImageExists`，串联默认 tag 补全、PullPolicy 预检（`imagePullPrecheck`）、凭据查找（`makeLookupPullCredentialsFunc` 组合 node keyring / external credential provider / pod secret）、退避重试（`flowcontrol.Backoff` + `prevPullErrMsg`）和 CRI 错误归一化（`evalCRIPullErr`）。
- `imagePuller` 接口（`puller.go`）：封装底层 `kubecontainer.ImageService.PullImage` 调用，提供 `parallelImagePuller`（token channel 限流）与 `serialImagePuller`（单 goroutine 队列）两种策略，由 `NewImageManager` 根据 `serialized` 参数选择。
- `ImageGCManager`（`image_gc_manager.go`）：维护 `imageRecords` 索引（支持 RuntimeClassInImageCRIAPI 特性下的 imageId+runtimeHandler 复合键），按 `ImageGCPolicy`（high/low threshold + min/max age）周期性触发 `GarbageCollect`，并通过 `PostImageGCHook` 暴露外部回调。

子包 `pullmanager`（`interfaces.go` + `image_pull_manager.go` + `fs_pullrecords.go` + `mem_pullrecords.go` + `noop_pull_manager.go`）在 FeatureGate `KubeletEnsureSecretPulledImages` 开启时介入，通过 `ImagePullManager` 记录拉取意图与已拉取凭据，避免重复拉取；其持久层由 `PullRecordsAccessor` 抽象，分别以文件系统（`fs_pullrecords.go`）与内存（`mem_pullrecords.go`）实现，并提供 `noop_pull_manager.go` 作为关闭时的空实现。

依赖方向：`images` → `pullmanager` → `credentialprovider` / `kubecontainer` / `cri-api`；GC 侧还依赖 `statsapi.FsStats` 提供的 ImageFS 用量信息。所有对外能力均通过接口（`ImageManager`、`ImageGCManager`、`imagePuller`、`PullRecordsAccessor`）暴露，便于测试注入与替换。
