模块以 `memory_manager.go` 中的 `Manager` 接口为对外边界，由内部 `manager` 结构体持有互斥锁、策略实例与 `state.State` 持久层，并依赖 `containermap.ContainerMap`、`topologymanager.Store`、`status.PodStatusProvider` 等外部组件。

- 策略抽象：`policy.go` 定义 `Policy` 接口（Name/Start/Allocate/RemoveContainer/GetTopologyHints/AllocatePod/GetAllocatableMemory），具体策略按文件名拆分：`policy_static.go`（Static，仅 Linux，独占+共享池+pod 级 bubble）、`policy_best_effort.go`（BestEffort，仅 Windows）、`policy_none.go`（None，无分配）。`NewManager` 通过 `policyType` 字符串选择策略，并在 Windows 上禁用 Static、在非 Windows 上禁用 BestEffort。
- 状态层：`state/` 子包提供 `State`/`Reader`/`writer` 三张接口，数据模型集中在 `state.go`（`NUMANodeMap`、`Block`、`ContainerMemoryAssignments`、`PodEntry` 等），持久化通过 `checkpoint.go` 实现的 V1/V2/V3 多版本 `checkpointmanager.Checkpoint` 双校验和格式写入磁盘文件 `memory_manager_state`，`state_checkpoint.go` 封装 `NewCheckpointState`，`state_mem.go` 提供内存态实现。
- 生命周期：`Start` 初始化 checkpoint → 调用 `policy.Start` 恢复/校验状态 → 计算各 NUMA 节点可分配内存；`AddContainer` 维护 container→pod 映射并清理已完成 init 容器的引用；`Allocate`/`AllocatePod` 委托给策略执行；`RemoveContainer` 通过 `containerMap` 反查 podUID/containerName 后调用策略释放；`removeStaleState` 在所有 source ready 后对比活跃 Pod 列表回收孤儿分配。
- 与 Topology Manager 集成：通过 `GetTopologyHints`/`GetPodTopologyHints` 返回 `topologymanager.TopologyHint`，使内存分配与其他资源控制器（CPU、hugepages）保持 NUMA 对齐。
