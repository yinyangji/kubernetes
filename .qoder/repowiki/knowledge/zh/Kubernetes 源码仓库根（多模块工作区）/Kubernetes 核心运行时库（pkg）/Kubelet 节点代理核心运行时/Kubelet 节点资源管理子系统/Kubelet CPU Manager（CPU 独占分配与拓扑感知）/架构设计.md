模块采用「管理器 + 策略 + 状态存储 + 拓扑发现」四层结构：
- 顶层 `cpu_manager.go` 暴露 `Manager` 接口，维护 `policy`、`state`、`containerMap`、`topology` 等字段，通过 `NewManager` 根据 `cpuPolicyName` 选择策略；`Start` 中加载 `state.NewCheckpointState` 持久化状态，并以 `wait.Until` 启动 `reconcileState` 周期循环。
- 策略层由 `policy.go` 的 `Policy` 接口定义（Allocate/RemoveContainer/GetTopologyHints/AllocatePod/GetAllocatableCPUs），当前提供 `policy_none.go`（无分配）和 `policy_static.go`（Guaranteed 整核独占，维护 SHARED/RESERVED/ASSIGNABLE/EXCLUSIVE 四池），`policy_options.go` 承载静态策略可选项。
- 状态层位于 `state/`，`state.go` 定义 `Reader`/`writer`/`State` 三张接口，`state_mem.go` 为内存实现，`checkpoint.go` 基于 `kubelet/checkpointmanager` 实现 V2/V3/V4 多版本 JSON 序列化与双校验和，用于 Kubelet 重启恢复。
- 拓扑层位于 `topology/`，从 cadvisor `MachineInfo` 解析出 `CPUTopology`（NUMA/Socket/Core/UncoreCache 层级），供静态策略做 SMT 对齐与 NUMA 亲和选择。
- 运行时交互：`reconcileState` 遍历活跃 Pod，对比 `lastUpdateState` 与当前 state，差异通过 `runtimeService.UpdateContainerResources` 写回 cgroup；`AddContainer`/`RemoveContainer` 维护 `containermap.ContainerMap` 以支持按 containerID 反向查找。
- 平台差异通过 `cpu_manager_others.go` 与 `cpu_manager_windows.go` 拆分，仅 Windows 有额外实现。
