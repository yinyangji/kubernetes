模块以 `selinux_warning_controller.go` 中的 `Controller` 为入口，遵循 KCM 标准 informer+workqueue 模式：监听 Pod/PVC/PV/CSIDriver 四类资源，通过 `common.AddPodPVCIndexerIfNotPresent` 建立 PVC→Pod 反向索引，将变更入队后由 `runWorker`/`processNextWorkItem` 串行处理。

内部按职责拆分为四个子包：
- `cache/`：`VolumeCache` 接口及 `volumeCache` 实现，维护 `volumes`（UniqueVolumeName → usedVolume）、`podToVolumes`（Pod → 卷集合）和 `conflicts`（卷 → 冲突列表）三张表，在 `AddVolume`/`DeletePod` 时计算对称冲突对；`GetPodsForCSIDriver` 用于 CSIDriver 的 `SELinuxMount` 字段变化时的批量重算。
- `translator/`：`ControllerSELinuxTranslator` 实现 `util.SELinuxLabelTranslator`，仅拼接显式 `SELinuxOptions` 字段而不读取宿主 `/etc/selinux`，并提供 `ConflictsParsed` 预解析版本供热路径复用。
- `internal/parse/`：纯函数 `ParseSELinuxLabel`，将 `user:role:type:level` 字符串切为 `[4]string`，被 cache 与 translator 共享。
- 顶层 `volumehost.go`：让 `Controller` 实现 `volume.VolumeHost` 与 `volume.CSIDriverVolumeHost`，但除 `GetKubeClient`、`CSIDriverLister`、`GetEventRecorder` 外其余方法均返回空/错误，因为本控制器只读不挂载。

依赖方向：controller → cache + translator + internal/parse + volume/util；metrics 通过 `RegisterMetrics` 注册 `StableCollector` 暴露 `selinux_volume_conflict` 指标，数据源直接拉取 cache 的当前冲突集。
