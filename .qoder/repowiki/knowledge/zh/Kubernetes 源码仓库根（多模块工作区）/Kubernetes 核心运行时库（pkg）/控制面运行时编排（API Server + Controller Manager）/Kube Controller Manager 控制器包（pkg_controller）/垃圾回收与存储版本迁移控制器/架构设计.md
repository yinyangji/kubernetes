本模块由四个相互独立的控制器包组成，均遵循 controller-manager 的 informer-listener-queue-worker 模式：
- `garbagecollector`：核心通用 GC。`GraphBuilder` 通过 `informerfactory.InformerFactory` 动态为所有可删除 GVR 创建 monitor（`graph_builder.go`），将事件聚合到单线程 `graphChanges` 队列中维护内存依赖图（`uidToNode`）；当叶子节点可删或 owner 需 orphan 时，分别入队 `attemptToDelete` / `attemptToOrphan`，由 `GarbageCollector.Run` 的多 worker 消费并调用 API server 执行删除/更新。`config/types.go` 暴露 `GarbageCollectorControllerConfiguration`（含 `EnableGarbageCollector`、`ConcurrentGCSyncs`、`GCIgnoredResources`），`metaonly` 子包仅保留 ObjectMeta 以最小化序列化开销。
- `podgc`：监听 Pod/Node informer，按阈值清理已终止 Pod、在 out-of-service 节点上强制删除 terminating Pod、清理孤儿 Pod 及 unscheduled terminating Pod，使用 `workqueue.TypedDelayingInterface[string]` 对 node 级做延迟重试。
- `storageversiongc`：监听 `apiserverinternal/v1alpha1.StorageVersion` 与 `coordination/v1.Lease`，当 apiserver lease 被删除时扫描并剔除对应 `ServerStorageVersion` 条目，两个独立 worker 分别处理 lease 删除与 storageversion 更新。
- `storageversionmigrator`：上层编排器。`SVMController` 监听 `storagemigration/v1.StorageVersionMigration`，`MigrationRunnerController` 负责同一 GR 下只激活一个 SVM（加 `MigrationRunning` condition），`ResourceVersionController` 驱动具体 CRD 的 stored version 迁移；SVMController 通过 `graphBuilder` 接口复用 garbagecollector 的 Monitor 来拉取待迁移对象的元数据。

依赖方向：`storageversionmigrator` → `garbagecollector.GraphBuilder`（仅读 Monitor），其余三个包彼此无依赖，均由 `controller.Interface` 抽象统一启动。
