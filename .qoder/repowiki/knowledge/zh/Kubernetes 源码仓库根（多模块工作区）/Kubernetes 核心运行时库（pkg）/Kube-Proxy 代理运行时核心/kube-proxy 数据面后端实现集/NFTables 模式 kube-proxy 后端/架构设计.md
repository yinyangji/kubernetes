单包 leaf 模块，通过 `//go:build linux` 约束仅在内核支持 nftables 时编译；非 Linux 平台由同名的空 `doc.go` 占位避免构建失败。
- 入口与双栈：`NewDualStackProxier` 分别构造 IPv4/IPv6 两个单栈 `Proxier`，再交由 `metaproxier.NewMetaProxier` 聚合，对外暴露统一的 `proxy.Provider` 接口（`proxier.go` 中 `_ proxy.Provider = &Proxier{}` 断言）。
- 运行时：每个 `Proxier` 持有 `knftables.Interface`、`conntrack.Interface`、`runner.BoundedFrequencyRunner`，通过 `Sync()` 入队、`syncProxyRules()` 实际写表；`setupNFTables()` 负责一次性创建 `kube-proxy` 表、base chains（filter/nat 各 hook）、jump chains 以及 hairpin/masquerade 等全局链。
- 数据平面结构：使用一组命名常量定义 base chain / jump chain / set / map 名称（如 `servicesChain`、`serviceIPsMap`、`noEndpointServicesMap`、`firewallIPsMap`），并通过 `nftElementStorage` 抽象对 nftables Set/Map 的 diff 写入，按 `containerType` 区分 set/map 并复用 `readOrReset` 流程。
- 事件驱动：实现 `OnServiceAdd/Update/Delete/Synced` 与 `OnEndpointSliceAdd/Update/Delete/Synced`，配合 `proxy.ServiceChangeTracker` / `proxy.EndpointsChangeTracker` 累积变更，仅在 `isInitialized()` 后触发 Sync，避免启动期部分状态污染。
- 依赖方向：本模块依赖上层 `pkg/proxy` 公共组件（`metaproxier`、`runner`、`healthcheck`、`metrics`、`util`、`conntrack`），反向不依赖任何上层；底层通过 `sigs.k8s.io/knftables` 操作内核 nftables，不直接 exec `nft` 命令。
- 清理：`cleanup.go` 提供 `CleanupLeftovers`，按 IP 族遍历并删除整个 `kube-proxy` 表，供进程退出或迁移场景调用。
