本模块按测试类型分层组织，各层独立可编译、可运行，并通过顶层入口或脚本编排：
- `test/cmd/`：基于 shell 的 kubectl 命令级集成测试，由 `legacy-script.sh` 按字母序 source 每个子文件（如 `apply.sh`、`apps.sh`），通过 `record_command` 输出 JUnit XML。
- `test/e2e/`：Ginkgo v2 端到端测试主入口 `e2e_test.go` 在 `TestMain` 中注册所有 feature 包（`apimachinery`、`apps`、`auth`、`network`、`storage`、`node`、`scheduling`、`windows` 等）并注入嵌入式 manifest/fixtures；`framework/` 提供 provider 无关的 `Framework` 结构体、命名空间隔离、指标抓取、超时控制等共享能力，被上层 suite 通过 `_ "..."` 导入以触发 ginkgo 发现。
- `test/e2e_dra/`、`test/e2e_kubeadm/`、`test/kubemark/`：针对 DRA、kubeadm 引导、kubemark 虚拟集群的专用 e2e 入口，各自维护独立的 suite 与启动脚本。
- `test/e2e_node/` 与 `test/e2e_node_windows/`：在宿主机上拉起 apiserver + kubelet 进程，直接验证 kubelet 行为，含 `services/`、`remote/`、`criproxy/`、`runner/` 等子系统。
- `test/integration/`：Go 单元测试风格的组件集成测试，按子系统（apiserver、scheduler、controllermanager、etcd 等）分目录，依赖本地 etcd/Docker。
- `test/conformance/`：从 Ginkgo spec 注释生成 conformance YAML 与文档的 standalone 工具链（`gen-conformance-yaml.sh`、`spec-to-docs.sh`、`walk.go`）。
- `test/declarative_validation/`：按 API Group 树形组织 declarative validation 用例，每个资源目录包含 `declarative_validation_test.go` 与 `zz_generated.validations.*_test.go` 生成文件，由 `coverage-allowlist.yaml` 控制覆盖率白名单。
- `test/images/`：测试用容器镜像集合（agnhost、busybox、nginx、sample-device-plugin 等），统一通过 `Makefile`/`image-util.sh` 构建，受 `.permitted-images` 约束。
- `test/fixtures/`、`test/utils/`：嵌入到二进制中的 YAML fixture 与通用辅助库（pki、pod store、ktesting 适配器等），供 e2e/integration 复用。
依赖方向自上而下：suite → framework → utils/fixtures/images，禁止反向引用。
