模块按「CLI 层」和「phases 逻辑层」两层组织，依赖方向单向：cmd/upgrade → phases/upgrade。

- cmd/kubeadm/app/cmd/upgrade：Cobra 命令入口。upgrade.go 注册 `kubeadm upgrade` 根命令并挂载 apply/plan/diff/node 四个子命令；common.go 提供 apply 与 plan 共享的 `enforceRequirements`（加载 UpgradeConfiguration、解析 --ignore-preflight-errors、构建 clientset、调用 preflight.RunRootCheckOnly 与 CheckClusterHealth、打印配置、返回 VersionGetter）；apply.go 基于 workflow.NewRunner() 组装升级工作流（PreflightPhase → ControlPlanePhase → UploadConfigPhase → KubeconfigPhase → KubeletConfigPhase → BootstrapTokenPhase → AddonPhase → PostUpgradePhase），并通过 applyData 结构体在阶段间传递 InitConfiguration、client、patchesDir 等运行时状态，同时支持 dry-run 模式通过 apiclient.NewDryRun() 注入 FakeDiscovery 与一系列 Reactor 模拟 API 行为。
- cmd/kubeadm/app/phases/upgrade：纯业务逻辑包，不依赖 Cobra。compute.go 定义 ClusterState / Upgrade 模型并通过 GetAvailableUpgrades 组合 VersionGetter 拉取各组件版本，计算可执行的 patch/minor/experimental/RC 升级路径；preflight.go 实现 CoreDNS 迁移预检（CoreDNSUnsupportedPlugins、CoreDNSMigration 两个 Checker）；versiongetter.go 封装离线/在线版本获取策略；policy.go/staticpods.go/postupgrade.go/health.go 分别负责升级策略判定、静态 Pod 替换、证书续期与集群健康检查；main_test.go 通过 pkiutiltesting.RunWithPrivateKeyFixtureDirectory 为测试准备 PKI fixture 目录。
- 边界：cmd 层仅做参数绑定与阶段编排，所有版本计算、组件探测、文件写入均委托给 phases/upgrade，从而让 apply 与 plan 两条分支复用同一套升级能力。
