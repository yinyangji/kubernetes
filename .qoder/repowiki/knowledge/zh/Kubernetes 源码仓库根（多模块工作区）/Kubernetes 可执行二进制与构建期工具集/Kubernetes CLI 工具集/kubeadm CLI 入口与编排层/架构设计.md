cmd/kubeadm 作为整个 kubeadm 二进制的主入口（kubeadm.go → app/cmd/cmd.go），通过 cobra 注册 init/join/reset/upgrade/token/certs/config/version 等顶层子命令；每个子命令再委托给 app/cmd/phases 下的“阶段”函数，由 phases/workflow 的 runner 按顺序执行。

- 配置契约：所有命令共享 apis/kubeadm/v1beta4→v1 的版本升级与默认值填充，以及 componentconfigs 中的 kubelet/kube-proxy 组件配置加载，确保同一份 YAML 在 init/join/upgrade 中语义一致。
- 能力复用：preflight、discovery、componentconfigs、shared_utilities（pkiutil、staticpod、runtime、apiclient、dryrun、image）被多个命令子树共同依赖，形成“命令层薄、能力下沉”的分层结构。
- 平台隔离：constants、preflight、componentconfigs/kubelet、initsystem、files 等目录通过 _unix/_windows 后缀文件实现 OS 差异，上层命令无需感知。
- 升级路径：upgrade_commands 独立于 init/join 的命令树，但复用 phases/workflow 与 shared_utilities 的 apiclient/dryrun/staticpod 等基础设施，保证 upgrade apply/node 与 init/join 的执行模型一致。
