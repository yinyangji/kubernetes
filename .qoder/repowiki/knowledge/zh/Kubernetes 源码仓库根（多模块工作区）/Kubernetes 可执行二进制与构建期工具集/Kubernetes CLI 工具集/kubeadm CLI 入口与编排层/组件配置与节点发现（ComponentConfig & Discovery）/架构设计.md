模块由两个并列子包组成：
- `componentconfigs`：以 `handler` + `configBase` 抽象出统一的 ComponentConfig 生命周期（Default/FetchFromCluster/FetchFromDocumentMap/Marshal/Unmarshal），通过全局 `known []*handler` 注册具体实现；每个组件（`kubelet.go`、`kubeproxy.go`）定义一个静态 handler 实例，并在 `scheme.go` 的 `AddToScheme` 中统一注入 runtime.Scheme，从而支持从 ConfigMap 反序列化为对应 GroupVersion 的对象；`checksums.go` 负责校验 ConfigMap 签名以区分“系统生成”和“用户自定义”。
- `discovery`：顶层 `For` / `DiscoverValidatedKubeConfig` 根据 `JoinConfiguration.Discovery` 选择三种后端——`token`（读取 cluster-info ConfigMap，用 JWS 签名 + CACertHashes 公钥固定做双向校验）、`file`（本地 kubeconfig 文件，必要时从 cluster-info 同步 CA）、`https`（远程 https URL 拉取 kubeconfig），最终返回可用于 TLS Bootstrap 的 `clientcmdapi.Config`。
依赖方向单向向外：本模块被上层 `join` 流程调用，不反向依赖 discovery/componentconfigs 之外的 kubeadm 应用层逻辑。
